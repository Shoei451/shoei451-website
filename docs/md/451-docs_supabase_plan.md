# Supabase設計・セキュリティ改善まとめ

## 1. 現状の問題

### ❌ アーキテクチャの問題

* フロントエンドから直接SupabaseのREST APIを叩いている
* anon keyを使って誰でもアクセス可能
* セキュリティをクライアント側に依存している

---

### ❌ パスワード認証の問題

* SHA-256のみ（ソルトなし・高速で総当たりに弱い）
* クライアント側でハッシュ生成・検証
* 「password_hashでフィルタ」＝防御になっていない

---

### ❌ RLSポリシーの致命的ミス

```sql
using (true)
```

→ 全ユーザーにSELECT許可
→ **全データ取得可能（実質公開状態）**

---

### ❌ セキュリティの本質的な誤解

* 「非表示」＝安全ではない
* 「パスワードがある」＝保護されているではない
* クライアントは信用できない

---

## 2. 現状のリスク

### 攻撃パターン

* API直接叩き → 全データ取得
* 総当たり攻撃（brute force）
* スクレイピング
* DevToolsでの解析

---

## 3. 設計の本質

### ❗重要な原則

* セキュリティはサーバー側でやる
* クライアントに秘密を置かない
* 「取得できない」状態を作る

---

### ❗役割分離（重要）

* 書き込み（管理）と読み取り（公開）は分離する
* DB常時依存を避ける設計が理想

---

## 4. 解決策の選択肢

---

### ✔ Option A：Supabase Auth + RLS（強）

**特徴**

* ログイン必須
* ユーザー単位で制御可能

**メリット**

* セキュリティが強い
* 正しい設計

**デメリット**

* UXが重い（アカウント作成が必要）

---

### ✔ Option B：Edge Function（最適バランス）

**特徴**

* ログイン不要
* サーバー側でパスワード検証

**構成**

```
クライアント → Edge Function → Supabase
```

**メリット**

* UX軽い
* セキュリティ改善
* DB直接アクセスを防ぐ

**デメリット**

* パスワード共有で突破される

---

### ✔ Option C：静的配信（出題部分）

**特徴**

* JSON化してCDN配信

**メリット**

* 高速
* Supabase停止の影響なし
* cron不要

---

## 5. 推奨構成（今回の最適解）

### ✔ 全体構成

```
[出題]
静的JSON（CDN）

[管理]
Supabase（問題追加）

[保護記事]
Edge Function経由で取得
```

---

### ✔ protected_posts の扱い

* DBに保存 → OK
* 直接アクセス → NG
* Edge Function経由 → OK

---

## 6. 改善ステップ

### Step 1（必須）

* 危険なRLS削除

```sql
drop policy "Allow all reads for password verification"
on public.protected_posts;
```

---

### Step 2

* Edge Function作成
* DBへの直接アクセスを遮断

---

### Step 3（余裕あれば）

* レート制限追加
* bcryptなどに変更

---

## 7. 設計思想（重要）

### ❗今回の最適バランス

* セキュリティ：中
* UX：高
* 実装コスト：低

---

### ❗本質的理解

* 完全防御は不要（用途的に）
* 「軽い制限」で十分
* ただし「穴だらけ」はNG

---

## 8. 結論

* 現状：危険（実質公開状態）
* 改善：Edge Functionが最適
* 方向性：

  * 「ログインなし」を維持
  * でも「サーバー側で制御」

---

## 9. 一言でまとめ

👉 クライアントで守るな。サーバーで守れ。
